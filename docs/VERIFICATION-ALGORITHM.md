## 1. 사용자 시나리오 테스트 (End User)

### 시나리오 A: 처음 사용하는 사용자

1. 앱 설치 후 첫 실행
2. 폴더 선택 → 스캔 시작 (아무 설정 건드리지 않음)
3. **확인**: "균형" 프리셋으로 스캔이 정상 실행되는가?
4. **확인**: 중복 그룹이 합리적으로 감지되는가? (명백한 중복을 놓치지 않는가?)
5. **확인**: 완전히 다른 사진이 중복으로 잡히지 않는가?

### 시나리오 B: 빠르게 훑고 싶은 사용자

1. 설정 > 스캔 > 프리셋에서 "빠른" 선택
2. 스캔 실행
3. **확인**: 속도가 "균형"보다 체감될 정도로 빠른가?
4. **확인**: 명백한 중복(동일 파일 복사본)은 잡아내는가?
5. **참고**: Stage 2 검증이 없으므로 일부 오탐 가능 — 이것이 허용 범위인가?

### 시나리오 C: 중요한 사진이라 신중하게

1. 설정 > 스캔 > 프리셋에서 "보수적" 선택
2. 스캔 실행
3. **확인**: 오탐(거짓 양성)이 거의 없는가?
4. **확인**: 약간 편집된 사진은 놓칠 수 있음 — 이것이 기대 동작인가?

### 시나리오 D: 편집된 사진도 찾고 싶은 사용자

1. 설정 > 스캔 > 프리셋에서 "정밀" 선택
2. 밝기/대비 조정, 크롭, 리사이즈된 사진이 포함된 폴더 스캔
3. **확인**: 편집된 버전도 중복으로 감지하는가?
4. **확인**: 스캔 시간이 합리적인가? (이중 검증이므로 가장 느림)

### 시나리오 E: 직접 조합하고 싶은 고급 사용자

1. 설정에서 알고리즘 토글을 직접 변경 (예: dHash만 ON + NMSE만 ON)
2. **확인**: 프리셋이 "사용자 정의"로 자동 전환되는가?
3. 슬라이더로 임계값 조정
4. **확인**: 변경한 조합으로 스캔이 정상 실행되는가?
5. 앱 재시작
6. **확인**: 사용자 정의 설정이 유지되는가?


### 4.2 TypeScript 타입 검증

```bash
npx tsc --noEmit
```

### 4.3 ScanEngine 모드 검증

| 모드 | 설정 | 예상 동작 |
|------|------|-----------|
| 단일 Hash + 단일 Verify | `['phash']` + `['ssim']` | 기존과 동일한 결과 |
| 복수 Hash + Union | `['phash', 'dhash']` + union | 두 해시 그룹 합집합 |
| 복수 Hash + Intersection | `['phash', 'dhash']` + intersection | 두 해시 그룹 교집합 |
| Hash만 (Verify 없음) | `['dhash']` + `[]` | Stage 1 결과 = 최종 결과 |
| 복수 Verify 순차 | `['ssim', 'nmse']` | SSIM 통과 → NMSE 추가 검증 |

### 4.4 DB 마이그레이션

| 검증 | 방법 |
|------|------|
| 기존 DB에 새 컬럼 추가 | 기존 settings.json 있는 상태로 앱 실행 → 에러 없음 |
| `option_algorithm_config` 저장 | 스캔 후 DB에 JSON 기록 확인 |
| 기존 스캔 기록 호환 | 이전 스캔 기록이 대시보드에 정상 표시 |

### 4.5 엣지 케이스

| 케이스 | 확인 |
|--------|------|
| 레거시 settings.json (phashThreshold/ssimThreshold) | 기본값으로 채워지는지 |
| enabledPlugins 필드가 남아있는 settings.json | 무시되고 정상 동작 |
| HEIC 이미지 | dHash/NMSE에서도 정상 처리 (sharpFromPath 사용) |

### 5.4 Stage 2 검증 효과

**pHash(8) + 각 SSIM 임계값:**

| SSIM | 그룹 수 | pHash only(8) 대비 |
|------|---------|-------------------|
| 없음 | 6 | 기준 |
| 0.75 | 3 | 3그룹 제거 (오탐 필터) |
| 0.80 | 2 | 4그룹 제거 |
| 0.82 | 2 | 4그룹 제거 (기본값) |
| 0.85 | 2 | 4그룹 제거 + 37/38/39 분할 |
| 0.90 | 1 | 5그룹 제거 |

### 5.5 Union vs Intersection 비교

**pHash(8)+dHash(8), Stage 2 없음:**

| 병합 전략 | 그룹 수 | 특성 |
|-----------|---------|------|
| Union | 8 | 넓은 커버리지 (어느 하나라도 유사하면 그룹) |
| Intersection | 4 | 정확 (두 해시 모두 유사해야 그룹) |

### 5.6 NMSE 검증 효과

**pHash(8) + NMSE:**

| NMSE 임계값 | 그룹 수 | pHash only(8) 대비 |
|-------------|---------|-------------------|
| 없음 | 6 | 기준 |
| 0.05 | 6 | SSIM보다 관대 |
| 0.10 | 7 | 더 관대 |

NMSE는 SSIM보다 필터 효과가 약함 — 단독보다 SSIM과 조합(순차 파이프라인)이 효과적.

## 6. 알려진 제약

- **Worker Threads**: stub 상태 — parallelThreads 설정이 실제 병렬 처리에 반영되지 않음.
- **복수 Hash 시 속도**: 각 파일에 대해 순차적으로 N개 해시 계산. 향후 Worker Threads로 병렬화 가능.
